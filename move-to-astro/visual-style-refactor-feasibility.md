# 全站视觉风格化重构规划

日期：2026-05-22  
项目：Mr.Xia 个人小站 Astro 版本  
状态：已吸收视觉落地回答，形成首轮实施报告

## 结论

将整个网站做一次风格化、完整级别的视觉重构是可行的，而且现在已经可以从“是否可行”推进到“按约束规划”。本轮回答明确了方向：网站应保持混合形态，整体接近中文互联网个人主页，参考 Fuwari 类博客的清爽、轻量、内容导向气质，但减少花哨装饰；当前暖色玻璃拟态不再延续；新视觉以中性色、统一色系、小圆角、适中内容密度、轻装饰背景、light / dark 双主题和便捷切换为核心。

仍然不建议一次性替换全站 CSS。当前 Astro 结构清楚，但 `src/styles/global.css` 仍是主要耦合点。推荐采用“设计系统先行、页面族群分批落地”的路线：先确定 token、主题、排版、圆角、组件边界、文案规则和文章阅读规范，再逐步处理首页、文章列表、文章正文、作品页、工具页、关于页、更新日志和 legacy 跳转兼容。

总体判断：

| 维度 | 评估 |
| --- | --- |
| 技术可行性 | 高 |
| 视觉决策复杂度 | 中，本轮已收敛主方向和首轮落地范围 |
| 实现复杂度 | 中高 |
| 回归风险 | 中高，主要来自全局 CSS、light / dark 主题和交互页面 |
| 推荐推进方式 | 多 phase、可回退、可阶段上线；首轮从全局外壳 + 首页开始 |
| 是否需要改内容 schema | 暂不需要 |
| 是否需要改 Worker / RSS / SEO | 暂不需要 |
| 是否需要新增 CI | 首轮暂不强制；若后续引入截图回归、主题快照或文件生成脚本，应补 workflow |

## 已确认规划约束

### 产品与内容定位

- 站点定位：混合形态，保留文章、作品、工具、关于的现有主导航顺序。
- 访客对象：同学朋友、技术同行、潜在合作方、自己日常使用和搜索引擎访客都存在，不偏向单一人群。
- 首页任务：首页就是入口页，核心职责是指引访问者进入文章、作品、工具和关于，不承担强营销或强叙事。
- 文章优先级：网站整体视觉效果与中文阅读体验同等重要，文章正文应保持博客阅读体验。
- 工具定位：工具页偏作品展示的一部分，不升级为主导航，也不做严肃生产力工作台。
- 最近更新：可以继续存在。
- 全站搜索：可以保留，但不是首页重点。
- 作品主推：不需要刻意突出某一个主项目。
- 关于页：暂时不增加个人介绍。
- 文章归档：定位为索引。
- 更新日志：可以顺带重构。
- Styleguide：内部维护，不需要公开强化。

### 首页已确认要求

- 当前时间组件保留。
- 首页主标题需要换掉，不继续使用“欢迎来到 Mr.Xia 的小站”。
- 新首页主标题必须处理不同视口宽度下的换行，不允许在常见宽度出现别扭断行、单字孤行或挤压。
- 首页不新增主推文章，保持当前不突出单篇文章的状态。
- 首页不放主推作品。
- 首页不直接强化工具快捷入口；工具仍归属作品体系，不作为首页重点入口。
- 首页 CTA 按内容模块自然出现，不强行规定固定数量。
- 首页是否加入滚动叙事视复杂程度决定，不为了动效而增加结构。
- 首页暂时不增加更强个人介绍。
- 最近更新作为索引保留，不改成时间线。
- 首页可以有动效，但不得因此加入解释型描述文案；所有文案仍遵循短、清楚、必要的原则。

### 文章体验已确认要求

- 文章列表形态定位为博客流。
- 分类和标签筛选保留。
- 搜索建议下拉体验保留。
- 无结果状态可以增强，例如提供清空筛选、换关键词或推荐最近文章。
- 桌面端文章目录继续右侧 sticky。
- 移动端目录保持现状：顶部折叠。
- 阅读进度条保留，但随整体风格重新视觉化。
- 阅读量保持现状，不额外强化。
- Giscus 评论是文章页唯一互动方式，必须保留并匹配新主题。
- 文章页返回入口顶部和底部各放一个。
- 文章元信息展示顺序保持现状。
- 不需要专门的长文模式。

### 工具页已确认要求

- 现有 tab 结构如果存在就保留，不为了改版重新发明导航。
- tab、工具标题和按钮不添加解释型描述语句。
- 计时器、随机选择器、Markdown 工具并列，不设单一最重要工具。
- Markdown 工具可以做得更接近编辑器界面，并可以加入编辑区与预览区同步滚动。
- 随机选择器不需要更强结果仪式感，只跟随整体视觉风格调整。
- 工具页输入内容使用 localStorage 保存即可。
- 不需要全屏或专注模式。
- 工具控件尺寸和密度保持中立，不刻意偏大或偏紧凑。
- 不需要键盘快捷键。
- 工具状态反馈保持中立，不额外追求强动效。
- Markdown 工具不需要共享文章 Markdown 正文样式。微信公众号/发布预览与站内文章是不同场景，应保持各自排版语境。

### 动效与交互已确认要求

- 当前页面转场保留，并可以在新视觉系统下进一步打磨。
- 按钮 ripple 可以保留，但需要纳入统一动效 token，避免与新风格冲突。
- 是否加入滚动触发、sticky 叙事或轻微深度效果保持中立，由页面复杂度和实际收益决定。
- 动效显眼程度保持中立，不刻意追求强存在感，也不完全隐藏。
- 哪些页面可以更有动效、哪些页面必须安静，由实现时根据页面任务决定：文章正文优先安静，首页和列表状态可以稍微有层级动效。
- 文章页切换可以保留方向感，用于增强前进/返回的空间关系。
- 工具页结果不需要额外动效，只保留必要状态反馈。
- 桌面端 hover 可以更丰富，移动端简化处理。
- reduced-motion 策略保持中立，但必须至少保证启用后不出现眩晕、阻塞或不必要的大幅运动。
- 页面 loading indicator 保留，并随新视觉统一。

### 可访问性与性能已确认要求

- WCAG AA 对比度尽可能满足，尤其是正文、按钮、链接、输入框、标签、代码块和 dark mode。
- 对低端设备性能的取舍保持中立，但不能为了装饰性效果造成明显卡顿。
- 是否避免外部字体和大型图片保持中立，实际选择时以加载成本和视觉收益判断。
- 暂时不做图片 AVIF/WebP 多尺寸处理，因为当前不计划引入重型图片资产。
- 支持 dark mode 的完整对比检查。
- 移动端最小支持宽度由实际布局确定，但必须覆盖常见手机宽度并避免横向滚动。
- 浏览器兼容以现代浏览器为准。
- focus ring 保留。
- 可以为了可访问性调整颜色和视觉层级。
- 视觉回归检查首轮先不急，等第一批页面稳定后再决定。

### 实施约束已确认要求

- 采用多 phase 推进，不做一次性无边界大改。
- 每个阶段可能需要独立上线，因此阶段边界要尽量保持可构建、可回退、可验证。
- 可以新增 CSS 文件结构，允许从单一 `global.css` 逐步拆分为更清晰的 token、base、layout、components、pages、article、tools 等层。
- 接受新增 npm 依赖，包括图标库、字体工具、截图测试或视觉回归相关工具；是否引入由收益、维护成本和构建影响决定。
- 不要求避免新依赖，但新增依赖必须有明确用途，不能为了“看起来高级”随意添加。
- 是否保留现有 class 名视情况而定：公共语义 class 可保留以降低回归，明显绑定旧视觉或阻碍重构的 class 可以替换。
- 接受重写部分页面结构。
- 接受重写首页和作品页文案，但必须继续遵守全局语言边界：短、直接、必要，不写解释型功能描述。
- 没有明确 deadline，可以慢慢来。
- 虽然不着急，但目标仍然是全站改造，而不是只做单个原型页后停止。

### 品牌与文案

- 品牌气质：直接、中立，不明显表达某种身份，不把学生、开发者、创作者标签放大。
- 参考方向：Fuwari 类博客，但更克制，不做得花哨。
- 站点气质：中文互联网个人主页，可适当轻松，但不能分散注意力。
- 副标题：可以加入，但必须短、清楚、服务识别。
- 前端文案硬约束：不要加入莫名其妙的说明文字，不写“点击这里可以……”“这里……了什么功能”这类解释型废话。页面文字只承担导航、命名、状态、必要约束和行动提示。

### 视觉语言

- 旧风格：现有暖色玻璃拟态作为历史版本，不再延续。
- 色彩：偏中性、统一色系，不保留现有橙、青绿、粉作为硬约束。
- 强调色：暂不定稿，先在全局外壳和首页中进行页面试色，或后续用其他 skill 继续探索。
- 主题：同时支持 light / dark 两套主题，并提供便捷切换。
- 默认主题：固定浅色为默认，深色通过手动切换进入，不默认跟随系统主题。
- 密度：中等内容密度。
- 圆角：整体小圆角。
- 卡片：按实际功能使用，禁止卡片套卡片和滥用卡片。
- 背景：保持一定装饰性，但不花哨。
- 首屏：暂时不做 full-bleed 改造，继续以站点主容器体系推进。
- 视觉素材：不要过分引入，暂时不需要 hero 图片、生成图、项目截图、地图或代码片段作为主视觉。
- 图标：尽量少用，只在主题切换等清晰功能中使用简单图标。
- 页面 logo：已有浏览器标签页 logo，页面上暂时不强化展示。

### 排版与内容呈现

- 中文字体：混合方向，可使用可靠、常见、质量稳定的字体，不使用奇怪小众字体。
- Web Font：可以接受，但需要谨慎选择并控制加载成本。
- 英文、数字、计时器和代码字体：视场景决定。
- 文章正文：博客风格，不改成杂志、论文或笔记系统。
- 标题：中立，不追求过大冲击。
- 正文行距：不需要刻意加宽。
- Markdown 扩展块：按现有 Markdown 渲染能力处理，后续只统一风格。
- 代码块：整体保持接近 GitHub 的清晰风格，但具体配色可根据新视觉调整。
- 文章图片：当前体验较满意，暂不大动，只随新风格调整。
- 标签和分类：保持现状层级。

### 作品页与 legacy 已确认要求

- legacy 页面跟现在一样做跳转，优先保证旧路径兼容和跳转正确，不把旧静态页面纳入新视觉覆盖主线。
- 作品页不需要截图或封面。
- 作品页不区分“主项目”和“小实验”。
- 项目标签、GitHub 链接、访问链接、APK 下载提示按实际内容和功能决定，不预先强行建立复杂规则。
- 作品页不需要时间线或状态字段。
- 暂时不需要为未来更多作品预留过滤或分类。
- 作品页继续承接工具集入口，保持当前归属关系。

### 文章发布流程已确认要求

- 正式文章发布以 Obsidian/R2 CLI 流程为主。
- `/new-post` 本地创作页纳入视觉重构，但不用过度打磨，只做轻量界面统一和可用性检查。
- `/new-post` 继续只在开发环境启用，并隐藏在公开导航之外。
- 本地创作页不需要草稿保存。
- Obsidian/R2 CLI 发布完全保持现状，包括目录命名、资源上传、`file/` 链接转换和输出文件名。
- 当前渲染 HTML 中的 tag 非空校验可以保留；publish 流程中的 tags 可以跳过，不要求非空。
- Markdown 工具明确保持公众号/HTML 工具定位，不与站内文章发布流程合并。
- 文章发布流程不需要额外前端说明；`/new-post` 只在本地使用，CLI 流程也不需要放到前端解释。

### 首轮视觉落地已确认要求

- 首批落地范围：全局外壳 + 首页。
- light / dark 默认策略：固定浅色为默认主题，允许用户手动切换；不默认跟随系统主题。
- 主题切换控件：桌面端和移动端分别设计，允许位置、尺寸和交互有所区别。
- 首页主标题由本报告先给 3 到 5 个候选，后续再选择或微调。
- 首页当前时间组件保留，但允许调整布局位置和视觉层级。
- 文章正文代码块不预先锁定现有 highlight.js 配色体系，可以根据新风格判断是否只改容器与 spacing，或同步调整配色。
- Giscus 的 light / dark 主题需要跟随站点主题自动切换，并尝试做视觉适配。

### 暂缓决策项

- 新视觉强调色暂缓定稿：可以先根据页面试色，或后续使用其他 skill 进行专门尝试。
- 视觉回归检查首轮先不急着加入，等第一批页面稳定后再决定。
- `styleguide.astro` 内部页首轮先不急着产出，待 token 和基础组件稳定后再评估。

## 当前站点观察

### 架构状态

当前项目已经是完整 Astro 静态站点结构：

- `src/layouts/BaseLayout.astro` 统一引入 `src/styles/global.css`，并挂载 Header、Footer、DynamicBackground、SkipLink、TransitionIndicator。
- `src/content.config.ts` 定义了 blog、works、tools、updates 四类内容集合。
- 主要页面包括 `src/pages/index.astro`、`src/pages/works.astro`、`src/pages/articles.astro`、`src/pages/articles/[...slug].astro`、`src/pages/works/tools.astro`、`src/pages/about.astro`。
- 工具类交互由 `src/components/ToolsSection.astro`、`TimerWidget.astro`、`RandomSelector.astro`、`MarkdownToolWidget.astro` 和 `src/scripts/` 下的脚本共同支持。
- 文章体验包含目录、阅读进度、标题锚点、图片灯箱、giscus 评论、KaTeX、阅读量代理等增强能力。

这意味着视觉重构主要可以集中在样式系统、布局组件、页面模板和交互状态，不需要重新迁移内容或重写站点核心数据流。

### 文章发布流程现状

当前文章发布相关能力分成三条边界不同的链路：

- Obsidian 到站点内容集合的 CLI 发布：`npm run publish -- --dry-run <obsidian-post-dir>` 和 `npm run publish <obsidian-post-dir>` 由 `scripts/publish-post.js`、`scripts/post-utils.js`、`scripts/markdown-utils.js` 负责。它读取 `OKP_VAULT` 下的文章目录，转换 `file/` 资源链接，写入 `src/content/blog/*.md`，并通过 R2/S3 客户端上传本地资源。
- 本地新建文章 API：`npm run api` 启动 `tools/api-server.js`，仅本地监听 `127.0.0.1:4322`，通过 `POST /api/new-post` 和 `NEW_POST_SECRET` 创建 `src/content/blog/*.md`。前端入口是开发环境才启用的 `src/pages/new-post.astro` 和 `src/components/NewPostForm.astro`。
- Markdown 工具：`src/components/MarkdownToolWidget.astro` 和 `src/scripts/markdown-renderer.ts` 面向 Markdown/HTML/微信公众号复制与预览。它属于作品体系下的工具，不等同于站内文章发布流程。

因此，对你反问“现有文章发布流程是否有对应修改计划”的回答是：有，但边界应很轻。CLI 是主流程，视觉重构只保证它不被破坏；`/new-post` 作为本地开发工具轻量纳入视觉统一，不作为本轮重点功能。

默认边界：

- 不默认修改文章 frontmatter schema。
- 不修改 Obsidian 目录命名、R2 上传、`file/` 链接转换、输出文件名、文章 slug、RSS、Worker 或 SEO。
- 不默认把 Markdown 工具并入站内文章发布流程，因为公众号预览和站内文章是两个场景。
- 可以在视觉重构中轻量统一 `/new-post` 本地创作页、表单、预览面板、状态提示和 Markdown 工具的界面风格。
- `/new-post` 继续只在开发环境启用并隐藏，不需要草稿保存。
- publish 流程允许 tags 为空或跳过；不要把 HTML 渲染工具的 tag 非空校验误扩散到 CLI 发布链路。
- 不需要新增前端说明页或内部发布说明文档。
- 如果后续要改实际写文件、发布、校验或上传逻辑，应作为独立 phase 推进，并同步补测试和 CI/CD 校验。

### 视觉现状

现有风格可以概括为：

- 暖色系玻璃拟态：橙、粉、青绿三色贯穿全站。
- 大圆角卡片系统：`card`、`shadowbox`、`page-intro`、`blog-card`、`filter-nav`、`editor-panel` 等大量共享玻璃面板样式。
- 渐变标题和柔和背景：站点背景、标题、按钮都依赖渐变和透明叠层。
- 全局动效：页面过渡、按钮 ripple、列表入场、文章转场、灯箱、loading spinner。
- 内容密度偏中等：页面结构清楚，但许多区域都以卡片承载，视觉节奏容易趋同。

当前 `src/styles/global.css` 约 3167 行，包含 token、base、布局、组件、页面、工具、文章、响应式和动画。它是全站视觉重构的核心风险点，也是最大的整理机会。

## 可行性拆解

### 1. 视觉换肤级重构

可行性：高  
风险：中  
适合目标：保持现有页面结构，只换颜色、字体、圆角、背景、阴影、按钮、卡片和动效节奏。

优点：

- 工作量较可控。
- 不影响内容集合、路由、SEO、Worker、RSS。
- 可以快速得到明显的新视觉。

限制：

- 如果继续沿用大量卡片和玻璃面板，风格变化可能只是“换皮”，很难达到“完全重构”的感受。
- 现有 CSS 耦合较强，直接替换变量不能解决所有细节。

### 2. 设计系统级重构

可行性：高  
风险：中高  
适合目标：重新定义全站视觉语言，包括 token、布局节奏、页面模板、组件状态、文章排版、工具界面和动效。

优点：

- 能真正建立可长期维护的视觉系统。
- 可减少重复样式和卡片泛滥。
- 后续新增页面更容易保持一致。

限制：

- 需要先做设计方向决策。
- 需要把 `global.css` 拆成更清晰的层级，例如 tokens、base、layout、components、pages、utilities。
- 需要逐页验证移动端和交互状态。

### 3. 信息架构加视觉重构

可行性：中高  
风险：高  
适合目标：不仅换视觉，还重新定义首页叙事、文章入口、作品展示、工具台层级和搜索方式。

优点：

- 最接近“全站重新设计”。
- 可以让个人站的定位更鲜明，比如偏作品集、偏文章刊物、偏工具实验室，或三者并重。

限制：

- 需要更多设计问题先定下来。
- 会影响页面文案、导航策略和用户路径。
- 更需要截图回归、移动端检查和内容边界测试。

## 推荐路线

采用“设计系统级重构”为主线，并做轻量的信息层级调整。首页只承担入口与方向指引，不做营销页；文章体系优先阅读体验；作品和工具保留现有归属关系；关于页保持简洁；更新日志随同作品体系做视觉统一。

推荐原因：

- 当前 Astro 架构已经足够支撑分阶段推进。
- 站点内容类型多，应该保持混合个人主页形态，不按单一 landing page 或单一博客模板处理。
- Fuwari 类博客的轻量内容组织值得参考，但需要降噪、减少装饰和避免过度可爱化。
- 文章阅读体验是核心验收项，工具页只需要保持可用并融入作品体系，不必升级为复杂工作台。
- `frontend-skill` 的“少卡片、短文案、克制动效”适合本项目，但首屏 full-bleed 和强视觉资产暂不采用。
- `ui-ux-pro-max` 的高优先级规则提示：light / dark 对比度、触控尺寸、响应式、性能、焦点状态和 reduced-motion 都必须作为基础验收。

## 已收敛设计方向

### 目标视觉 thesis

一个中性、直接、轻装饰的中文个人主页：保留博客的清爽阅读和个人站的入口感，用小圆角、统一中性色、适度留白、克制动效和手动 light / dark 主题切换建立稳定气质。首轮从全局外壳和首页开始，先把导航、主题切换、背景、首页标题和时间组件的基调定住。

### 风格边界

采用：

- 中文互联网个人主页的亲切结构。
- Fuwari 类博客的轻量、清爽、内容导向感。
- 中等内容密度。
- 小圆角、轻边框、低噪声背景。
- 中性色基础上的单一克制强调色；强调色暂不定稿，先以页面试色为准。
- 卡片只用于确实需要框定的内容或交互。

避免：

- 通用 SaaS。
- 二次元。
- 赛博朋克。
- 极简黑白。
- 过度玻璃拟态。
- 强科技蓝。
- 营销落地页。
- 卡片套卡片。
- 解释型、教学型、堆功能说明的前端文案。

### 页面族群策略

首页：

- 保持首页作为入口页。
- 导航顺序维持文章、作品、工具、关于。
- 当前时间组件保留，但允许调整布局位置和视觉层级。
- 首页主标题更换，并专门处理响应式换行。
- 最近更新保留为索引，不做时间线。
- 搜索可保留但降级为辅助入口。
- 不新增主推文章、主推作品或强化工具快捷入口。
- CTA 按内容自然出现，不为凑布局强行增加按钮。
- 不做 full-bleed，不引入重型 hero 图。
- 可以有克制动效，但文案短，避免描述“这里有什么功能”。

文章：

- 文章列表采用博客流，继续承担博客索引和全站搜索功能，但视觉上降低工具感。
- 分类、标签筛选和搜索建议保留。
- 无结果状态可以更有帮助。
- 正文页保持博客阅读体验，控制行宽、层级、代码块和图片样式。
- 桌面端目录继续右侧 sticky，移动端目录保持顶部折叠。
- 阅读进度条保留但换风格，阅读量和元信息顺序保持现状。
- 返回入口顶部和底部各放一个。
- Giscus 评论保留，是唯一互动方式。
- 图片灯箱和现有图注体验只做风格同步，不大改交互。
- 分类、标签、阅读信息保持现状层级。
- 文章发布流程不随阅读页视觉改变；CLI 是主流程，`/new-post` 只作为开发环境隐藏工具做轻量界面统一。

作品：

- 不强推单个作品。
- 保持作品页作为项目列表和工具集入口。
- 项目条目可从大圆角玻璃卡片改为更轻的档案条目、列表块或单层面板。
- 不加入截图/封面，不区分主项目和小实验，不做时间线或状态字段。
- 项目标签、外链、下载提示按实际功能决定，不强行做复杂展示体系。
- 更新日志可顺带纳入同一风格。

工具：

- 不进入主导航。
- 保持作品体系下的工具展示。
- 不做复杂 dashboard，也不做重型工作台。
- 现有 tab 保留，三个工具并列。
- Markdown 工具可增强为更像编辑器，并支持编辑区/预览区同步滚动。
- 工具输入使用 localStorage 保存。
- 不做全屏/专注模式，不加键盘快捷键。
- 随机选择器不加结果仪式感。
- Markdown 工具不共享文章 Markdown 正文样式，也不默认并入站内文章发布流程。
- 重点是输入、按钮、tab、结果和预览区稳定、清楚、可用。

关于：

- 暂时不增加个人介绍。
- 保留法律、隐私、免责声明和联系方式。
- 视觉上轻量，不做大段解释。

这条路线能避免当前“全站都是柔和玻璃卡片”的重复感，也不会把网站推向过度风格化、过度解释或过度产品化。

## 候选内容计划与交互 thesis

基于本轮回答，内容计划不追求强叙事或强营销，而是让首页、文章、作品、工具和关于各自完成清楚的入口职责。首轮实现从全局外壳和首页开始；文章发布流程保留现有逻辑，以 CLI 为主，`/new-post` 只做轻量视觉纳入。

内容计划：

- 首页：保留入口页定位，保留当前时间组件但允许调整位置；替换主标题并处理响应式换行；按文章、作品、工具、关于建立清楚路径；最近更新作为索引保留，搜索降级为辅助能力。
- 文章：文章列表以博客流承载内容索引，保留分类/标签筛选与搜索建议；正文页专注阅读、目录、评论和图片查看；本地新建文章页纳入轻量视觉统一，不改发布逻辑。
- 作品：保持作品集合，不强推单个项目；用更轻的项目档案样式替代重复大卡片；不加入截图/封面、主次项目、时间线、状态字段或未来过滤系统。
- 工具：保留在作品体系下，视觉上作为作品的一部分；三个工具并列，保留 tab；Markdown 工具可增强编辑体验和同步滚动，但仍保持公众号/HTML 预览语境。
- 关于：暂时只保留必要站点说明、法律信息和联系方式。

交互 thesis：

- 主题切换是第一优先级交互：默认固定浅色，用户可手动切换 light / dark；切换要可见、稳定、记忆用户选择，并避免闪烁。
- 主题切换控件在桌面端和移动端分别设计，不要求完全相同的位置或形态。
- 动效只强化状态变化：搜索、筛选、tab、工具状态、图片灯箱可以有轻微过渡，长文阅读区域保持克制。
- 页面转场保留并可进一步打磨，文章页切换可以保留方向感。
- 按钮 ripple 和 loading indicator 保留，但必须统一到新动效 token。
- 桌面 hover 可以更丰富，移动端简化；工具结果不额外加仪式感动效。
- 滚动触发、sticky 叙事或轻微深度效果保持中立，只在确实改善层级和体验时采用。
- 首屏不做 full-bleed，也不引入重型图片；通过排版、间距、背景纹理和导航层级建立识别。
- 工具页 localStorage 保存属于实用增强，不应配套增加说明段落；状态反馈中立即可见即可。
 
可访问性与性能 thesis：

- 可读性和对比度优先于装饰，WCAG AA 尽可能满足。
- dark mode 不只做反色，需要单独检查文字、边框、按钮、代码块、评论、灯箱和表单。
- 以现代浏览器为准，保留 focus ring，允许为了可访问性调整色彩和层级。
- 图片资产暂不做 AVIF/WebP 多尺寸管线，后续若引入重型图片再单独规划。
- 视觉回归检查暂缓，不作为首轮必备项；后续第一批页面稳定后再决定是否加入。

## 文章发布流程规划入口

这部分不应该被全站视觉重构顺手“隐性改掉”。已确认官方主流程是 Obsidian/R2 CLI，`/new-post` 是开发环境隐藏工具，Markdown 工具保持公众号/HTML 工具定位。

当前建议：

- 视觉重构主线只保证文章发布链路不被破坏。
- `/new-post` 本地创作页可以随全站样式做轻量视觉统一：表单、编辑区、预览区、状态反馈、错误提示、按钮层级和移动端布局都纳入基础检查，但不用作为重点页面精修。
- `/new-post` 的页面文案需要压短，避免“通过本地 API 创建新的博客 Markdown 文件”这类前端解释句出现在新视觉中；前端界面只保留必要标签和状态。
- `/new-post` 继续只在开发环境启用，并隐藏在公开导航之外。
- `/new-post` 不做草稿保存。
- Obsidian/R2 CLI 发布保持当前命令、环境变量、目录命名、资源上传、`file/` 链接转换和输出文件名。
- Markdown 工具保持工具页语境，不直接复用文章正文样式，也不默认承担站内发布。
- publish 流程中的 tags 可以跳过，不要求非空；HTML 渲染工具中的 tag 非空校验不迁移到 CLI 发布链路。
- 不需要新增前端发布说明，也不需要把 CLI 流程单独整理成前端说明文档。
- 如果后续要改 `createPostFile`、`buildPublishPlan`、R2 上传、payload 校验或文件命名，需要同步补测试。
- 按仓库约束，凡是修改文件操作功能，都应同步补或更新 `.github/workflows/` 下的 CI/CD 校验，覆盖正常写入、空输入、缺文件、重复文件和 JSON/frontmatter 格式检查。

可选执行切分：

1. 发布链路固化：Obsidian/R2 CLI 是主流程；`/new-post` 是本地开发辅助；Markdown 工具是公众号/HTML 工具。
2. 本地创作界面轻量视觉统一：不改写文件逻辑，只改布局、状态、可访问性、移动端和 light / dark。
3. 发布体验增强暂不纳入：不做草稿保存，不整理 dry-run 输出，不新增前端说明。
4. 文件操作与 CI：一旦改变生成或写入逻辑，补充 Node 测试和对应 workflow。

粗略工作量预估：

| 范围 | 预估 |
| --- | --- |
| 只做视觉换肤 | 2 到 4 个工作日 |
| 设计系统级重构 + light / dark | 7 到 11 个工作日 |
| 视觉 + 轻量信息层级调整 + 截图回归 | 10 到 14 个工作日 |
| 全站多 phase 完整改造 | 12 到 18 个工作日 |

以上估算默认单人推进，且不包含大量新增插图、摄影、复杂动画或视觉资产管线。light / dark 已进入规划范围，因此需要在每个阶段同步验收两套主题。由于目标是全站改造，不建议只做一个原型页后停住；可以先做一个代表性页面作为校准点，但后续要进入全站分阶段落地。

## 分阶段实施规划

### 阶段 0：页面审计与首轮范围固化

目标：

- 固化首轮范围：全局外壳 + 首页。
- 将强调色、视觉回归检查、`styleguide.astro` 首轮建设作为暂缓项记录。
- legacy 页面按当前方式保持跳转，阶段 0 只确认旧路径清单和跳转完整性。
- 确认作品页维持无截图/封面、无主次项目、无时间线/状态字段、无未来过滤系统的边界。
- 固化文章发布流程边界：CLI 为主，`/new-post` 为开发环境隐藏辅助，Markdown 工具为公众号/HTML 工具。
- 建立页面清单和状态清单：空状态、加载、搜索结果、无结果、错误、评论、图片灯箱、移动端导航、工具 tab、表单状态。
- 建立多 phase 范围切分，明确每个阶段是否可独立上线、如何回退、如何验证。
- 确认新增依赖候选清单，例如图标库、字体、截图测试或视觉回归工具。

产出：

- 已确认问题记录。
- 全站页面重构范围表。
- legacy 跳转清单和验证方式。
- 文章发布流程保留现状、轻量纳入 `/new-post` 视觉统一的结论。
- 多 phase 执行切分。
- 新增依赖候选与取舍理由。
- 首页主标题候选和响应式换行检查要求。
- 不改动范围说明，例如文章图片交互、分类标签层级、关于页内容结构等。

### 阶段 1：设计系统与主题基础

目标：

- 定义 light / dark 双主题 token：背景、文字、边框、强调色、链接、代码块、按钮、输入框、状态色。
- 默认主题固定为浅色；深色由用户手动切换，不默认跟随系统主题。
- 定义字体栈：中文混合字体、英文/数字/代码字体；外部字体只选可靠常见字体，并控制加载。
- 定义小圆角、轻边框、低阴影、间距、z-index、动效时长。
- 定义卡片使用规则：只允许单层容器，不使用卡片套卡片。
- 定义装饰背景规则：低噪声、可降级、两套主题都可读。
- 定义主题切换策略：桌面端和移动端分别设计按钮位置、图标、localStorage 记忆和首屏防闪烁。

建议 CSS 分层：

- `tokens.css`
- `base.css`
- `layout.css`
- `components.css`
- `article.css`
- `tools.css`
- `pages.css`
- `utilities.css`

是否拆文件可以根据实现阶段决定，但即使仍由 `global.css` 汇总，也应该先建立清晰的逻辑层。

实施约束：

- 接受新增 CSS 文件结构，优先从 token 和主题层开始拆。
- 是否保留旧 class 名按风险判断，不能为了保留 class 牺牲新系统边界。
- 如果新增依赖，需要在本阶段记录用途、替代方案和移除成本。

### 阶段 2：站点外壳与基础组件

目标：

- 重构 Header、Footer、DynamicBackground、SkipLink、TransitionIndicator。
- 添加或改造主题切换控件，桌面端和移动端分别适配。
- 优化导航在桌面和移动端的表达，维持文章、作品、工具、关于顺序。
- 统一页面转场、按钮 ripple、loading indicator 和 reduced-motion 行为。
- 建立基础按钮、链接、tag、badge、输入框、面板、列表和分隔线样式。

重点：

- Header 不应该抢走太多首屏预算。
- 移动端导航触控目标应不小于 44px。
- 焦点状态必须保留且清晰。
- 不在页面 logo 上做额外强调。
- 避免把说明性文案写进组件。
- 桌面 hover 可以更丰富，移动端需要简化。
- 文章页面转场可以保留方向感。

### 阶段 3：首页和内容入口

目标：

- 保持首页作为站点入口页。
- 保留当前时间组件。
- 允许调整当前时间组件的布局位置和视觉层级。
- 替换首页主标题，并为不同视口宽度设计稳定换行。
- 减少重复 CTA 和说明文字。
- 让文章、作品、工具、最近更新之间有更清晰的层级。
- 保留最近更新作为索引，搜索降级为辅助入口。

可能改动：

- 首页 hero 从大圆角玻璃卡片改为小圆角或无卡片的入口区域。
- 最近更新可以从卡片流改成更轻的列表或索引，不做时间线。
- 四个入口保持当前优先级，不新增工具主导航。
- 不添加主推文章、主推作品或首页工具快捷重点区。
- 首页文案遵循短标签和短句，不写功能解释段落。
- 首页动效视复杂程度决定，只允许服务层级、进入状态或轻微反馈。

### 阶段 4：文章体系

目标：

- 重构文章博客流、筛选、搜索建议、无结果状态、归档页。
- 重构文章正文排版风格、目录、代码块、引用、表格、KaTeX、评论区。
- 文章图片与灯箱只做风格同步，不大改交互。
- 顶部和底部都提供返回入口。

重点：

- 中文阅读行宽要稳定。
- 标题层级要清楚。
- 目录交互保持桌面右侧 sticky、移动端顶部折叠。
- 阅读进度条保留并换成新风格。
- 阅读量、文章元信息顺序保持现状。
- 图片灯箱和文章内容不能互相破坏层级。
- Giscus iframe 主题需要匹配新视觉，作为唯一互动方式必须稳定，并跟随站点 light / dark 自动切换。
- 代码块采用接近 GitHub 的清晰风格，但配色和容器可根据新视觉判断。
- 分类和标签保持现状层级，不抢正文注意力。
- 不增加长文模式。

### 阶段 4.5：文章发布与本地创作工具

目标：

- 作为轻量插入阶段，不阻塞文章阅读页视觉重构。
- `/new-post` 只做视觉统一和基础可用性检查，不做重点精修。
- 范围限定为表单布局、编辑/预览面板、按钮、状态提示、错误提示、移动端、light / dark 和可访问性。
- 不做草稿保存、slug/frontmatter 预览、重复文件提示、dry-run 输出整理或新增发布说明。
- 保持 Obsidian/R2 CLI 发布逻辑稳定。
- 保持 Markdown 工具与站内文章发布流程分离。

重点：

- 不能在视觉重构中悄悄修改内容 schema、R2 上传、资源链接转换、文章 URL 或 Worker/RSS/SEO 行为。
- 前端界面不写解释型文案。
- `NEW_POST_SECRET` 不应被 localStorage 保存；本轮不做草稿保存，因此不新增相关本地存储。
- publish 流程中的 tags 可以为空或跳过，不把 HTML 渲染工具的 tag 非空校验扩散到 CLI。
- 如果改动文件写入、发布计划、payload 校验或上传逻辑，需要补充对应测试。
- 如果实现文件操作功能改动，需要按仓库规则同步补 CI/CD workflow。

### 阶段 5：作品和工具体系

目标：

- 作品页从重复大卡片升级为轻量项目档案。
- 不添加作品截图或封面。
- 不区分主项目和小实验。
- 不添加时间线、状态字段或未来过滤系统。
- 项目标签、GitHub 链接、访问链接、APK 下载提示根据实际功能保留或弱化。
- 工具集入口继续由作品页承接，保持当前归属关系。
- 工具页保持作品体系下的展示和使用，不做复杂工作台。
- Timer、Random Selector、Markdown Tool 并列展示。
- 保留现有 tab 结构。
- Markdown 工具增强为更接近编辑器的界面，并支持编辑区/预览区同步滚动。
- 工具输入使用 localStorage 保存。
- Timer、Random Selector、Markdown Tool 的控件、状态、按钮、输入区和预览区统一视觉语言。
- 顺带统一更新日志页面视觉。

重点：

- 工具界面不做大型营销 hero，也不升级为独立产品页。
- 输入框、按钮、tab、结果区需要稳定尺寸，避免交互时布局跳动。
- Markdown 工具预览样式不直接共享文章正文样式；微信公众号/发布预览与站内文章排版分开维护。
- 随机选择器不增加结果仪式感，只跟随视觉风格变化。
- 不做全屏/专注模式，不加键盘快捷键。
- 控件尺寸和状态反馈保持中立。
- 项目条目避免卡片套卡片。
- 不刻意突出单个主项目。
- 不用截图、封面或时间线制造额外视觉负担。
- 不为 tab 或工具说明增加描述性语句。

### 阶段 6：legacy 跳转与辅助页面

目标：

- legacy 页面跟现在一样做跳转。
- 验证 `public/` 中旧路径、迁移前入口和兼容文件没有被新视觉改造破坏。
- 不默认对旧静态页面做视觉同步。
- 如果审计中发现旧页面仍直接承载有效内容，再单独确认迁移或轻量覆盖，不并入本轮默认范围。
- 内部 `styleguide.astro` 首轮暂缓，不作为全局外壳 + 首页落地的前置条件；待 token 和基础组件稳定后再评估。

重点：

- 不把 legacy 页面拖入主线视觉重构，避免范围失控。
- legacy 验收重点是跳转、链接和兼容路径，不是新视觉一致性。

### 阶段 7：质量验证

最低验证：

- `npm run build`
- `npm test`
- 桌面宽度检查：首页、文章列表、文章正文、作品页、工具页、关于页、更新日志。
- 移动宽度检查：以实际布局确定最小宽度，至少覆盖常见手机宽度，并重点检查 375px、480px、768px。
- light / dark 两套主题都检查。
- dark mode 对比度单独检查。
- 检查键盘焦点、搜索建议、工具 tab、评论、图片灯箱、返回链接。
- 检查 reduced-motion。
- 检查浏览器 console 没有新增错误。
- 以现代浏览器为主进行验证。

后续可选增强：

- 增加 Playwright 截图检查或视觉回归检查。
- 建立 visual regression 基准。
- 为 styleguide 页面补完整 token 和组件状态。

### 阶段 8：分阶段上线与收口

目标：

- 每个 phase 完成后保证可构建、可回退、可独立验收。
- 如果阶段允许独立上线，记录上线范围和剩余未覆盖页面。
- 全站改造完成后做一次整体视觉一致性审计。

重点：

- 不因“慢慢来”而只停留在原型页。
- 阶段边界应围绕用户可见页面族群，而不是随意按 CSS 文件拆。
- 每个阶段都保留全局语言边界，不在后续补文案时引入解释型句子。

## 风险矩阵

| 风险 | 等级 | 原因 | 缓解方式 |
| --- | --- | --- | --- |
| 全局 CSS 牵连过大 | 高 | 多页面共用 `card`、`shadowbox`、`page-intro` 等 class | 先抽 token 和组件层，再逐页面替换 |
| light / dark 不一致 | 高 | 双主题需要同时覆盖背景、文字、边框、代码块、表单、giscus、灯箱等 | 主题 token 先行，每阶段同时验收两套主题 |
| 文章正文回归 | 高 | Markdown 内容、代码、表格、图片、KaTeX、目录、灯箱都依赖样式 | 建立文章正文样例页和长文测试清单 |
| 卡片继续滥用 | 中高 | 现有样式语言高度依赖卡片和玻璃面板 | 明确单层容器原则，优先列表、分隔线、网格和留白 |
| 工具页可用性下降 | 中高 | 视觉化容易影响输入、按钮、tab 和预览区效率 | 工具页保持作品展示定位，但控件和状态必须稳定 |
| 误改文章发布流程 | 高 | 发布链路涉及文件写入、R2 上传、资源链接转换和本地 API | CLI 主流程保持现状，`/new-post` 只做轻量视觉统一 |
| 文件操作改动缺少 CI | 高 | 仓库约束要求文件操作功能配套自动化校验 | 改 `createPostFile`、发布计划或上传逻辑时同步补测试和 workflow |
| 本地创作页暴露敏感信息 | 中高 | `/new-post` 使用 `NEW_POST_SECRET`，不应被状态持久化误存 | 不做草稿保存，也不保存 secret |
| 首页标题响应式断行难看 | 中 | 新标题如果只按桌面设计，移动端可能出现孤字、挤压或过短行 | 在 375px、480px、768px 和桌面宽度单独检查标题换行 |
| Markdown 工具与文章样式混同 | 中 | 发布预览和站内文章是不同场景，复用正文样式会造成语义混乱 | 工具预览和文章正文分开定义 token 与排版规则 |
| localStorage 状态处理不一致 | 中 | 工具输入保存如果没有统一策略，可能产生过期、空值或跨 tab 行为差异 | 给工具输入保存建立统一 key、初始化、清空和错误处理策略 |
| 移动端首屏拥挤 | 中高 | Header、hero、CTA、时间组件可能挤占首屏 | 先定义移动端首屏预算 |
| 动效过度 | 中 | 当前已有页面转场、ripple、reveal、灯箱，且计划保留并打磨 | 统一 motion token，文章正文保持安静，工具结果不额外加仪式感动效 |
| reduced-motion 处理不足 | 中 | 用户对 reduced-motion 保持中立，但体验不能因动效产生眩晕或阻塞 | 至少关闭大幅位移、强滚动动效和非必要循环动画 |
| 桌面 hover 与移动端行为不一致 | 中 | 桌面可以更丰富，移动端需要简化 | hover 只作为增强，不承载必要信息或唯一操作 |
| 性能变差 | 中 | 双主题、字体、背景纹理和动效都可能增加负担；视觉回归若后续加入也会增加维护成本 | 字体谨慎、减少 blur 和大面积阴影，避免重型视觉资产 |
| 新依赖膨胀 | 中 | 已接受新依赖，可能引入维护成本、构建时间和样式体积增长 | 每个依赖必须记录用途、替代方案和移除成本 |
| phase 边界失控 | 中高 | 多 phase 推进如果边界不清，容易出现半新半旧且难回退 | 每个 phase 定义页面范围、验收命令、回退策略和可上线状态 |
| 旧 class 与新系统冲突 | 中 | 旧 class 名可能绑定旧视觉，保留过多会拖累重构 | class 保留按风险判断，旧视觉强绑定 class 可替换 |
| dark mode 对比不足 | 中高 | 深色主题不是简单反色，代码块、评论、灯箱、表单都可能出问题 | 单独检查 dark mode 对比度，必要时调整色彩层级 |
| 文案变啰嗦 | 中 | 重构时容易加入解释型提示和功能描述 | 建立前端文案约束，只保留导航、状态、必要提示和行动 |
| 品牌方向摇摆 | 中 | 本轮已收敛方向，但未回答问题仍可能影响细节 | 把未回答问题限定在页面和交互层，不再重开主风格 |
| 可访问性下降 | 中高 | 低对比、隐藏焦点、图像文字叠加都有风险 | 以 WCAG 对比、键盘导航、触控尺寸作为硬约束 |
| legacy 跳转失效 | 中 | `public/` 中仍有旧 URL 和兼容入口 | 保持现有跳转策略，阶段验收旧路径和链接 |

## 设计验收标准

一次成功的全站视觉重构应该满足：

- 访客在 5 秒内能看出这是 Mr.Xia 的个人站，而不是通用模板。
- 首页、文章、作品、工具四类页面有同一套视觉 DNA，但各自任务清楚。
- 首页主标题在常见视口宽度下换行自然，不出现孤字、挤压或不协调断行。
- 文章正文比现在更好读，而不是更花。
- 文章列表是清楚的博客流，筛选、搜索建议、无结果状态都保持可用。
- 文章目录、阅读进度、阅读量、Giscus 和返回入口符合已确认行为。
- 文章发布流程没有被视觉重构隐性改变；Obsidian/R2 CLI 是主流程，本地 `/new-post` API 是开发环境隐藏辅助，Markdown 工具是公众号/HTML 工具。
- `/new-post` 纳入轻量视觉范围，保持本地开发环境入口属性，界面更清楚，但不做草稿保存、不暴露或保存密钥。
- publish 流程允许 tags 跳过；HTML 渲染工具里的 tag 非空校验不影响 CLI 发布。
- 工具页比现在更稳定，同时仍像作品体系的一部分。
- 工具页保留 tab，三个工具并列；Markdown 工具支持更好的编辑体验与同步滚动；输入可通过 localStorage 保存。
- Markdown 工具预览和站内文章正文保持场景区分。
- 作品页不通过截图/封面、主次项目、时间线、状态字段或过滤系统增加额外复杂度。
- legacy 页面保持当前跳转兼容，不要求同步新视觉。
- 全站减少无意义卡片，不出现卡片套卡片，保留真正需要承载交互或内容组的面板。
- light / dark 两套主题都完整、可读、可便捷切换。
- dark mode 经过独立对比度检查，正文、链接、按钮、输入框、代码块、评论和灯箱都可读。
- 页面转场、按钮 ripple、loading indicator 保留并统一新风格，文章切换方向感可用且不突兀。
- 桌面 hover 是增强体验，移动端简化后仍能完整操作。
- reduced-motion 下没有大幅位移、强滚动动效或阻塞性动画。
- 前端文案没有无意义解释，不出现“点击这里可以……”一类句子。
- 移动端没有横向滚动、遮挡、文字挤压或按钮过小。
- 键盘焦点、aria 状态、skip link、focus ring 不退步。
- WCAG AA 尽可能满足，必要时允许调整颜色和层级。
- 以现代浏览器为主验证。
- 视觉回归检查首轮暂缓；如果后续引入，应覆盖首页、文章列表、文章正文、作品页、工具页和 dark mode 关键状态。
- 多 phase 推进时，每个阶段都有明确范围、验收方式、回退方式和可上线判断。
- 新增依赖都有明确用途，不引入无意义装饰依赖。
- 如果改动文件操作功能，必须有对应测试和 CI/CD 校验。
- 页面结构和文案可以重写，但全局语言边界不被破坏。
- `styleguide.astro` 首轮暂缓；后续若启用，应能清晰展示和维护新样式。

## 暂缓决策项

基于目前已回答内容，报告已经可以支撑首轮“全局外壳 + 首页”落地。以下三项暂时搁置，不作为首轮实施前置条件。

1. 新视觉强调色：先根据页面试色，或后续使用其他 skill 进行专门尝试。
2. 视觉回归检查：首轮先不急，等第一批页面稳定后再决定。
3. `styleguide.astro` 内部页：首轮先不急，待 token 和基础组件稳定后再评估。

已确认边界：首轮从全局外壳 + 首页开始；默认固定浅色，手动切换 light / dark；主题切换桌面端和移动端分别设计；首页主标题先由本报告提供候选；当前时间组件允许调整布局；代码块配色视情况决定；Giscus 跟随站点主题自动切换。多 phase 推进、可能阶段上线、接受新增 CSS 结构和依赖、class 名视情况保留、可重写结构和文案、无 deadline、目标仍是全站改造；legacy 页面保持当前跳转；作品页不加截图/封面、主次项目、时间线、状态字段或未来过滤系统；文章发布以 CLI 为主，`/new-post` 仅开发环境隐藏且轻量纳入视觉统一，不做草稿保存，不改 CLI 流程。
