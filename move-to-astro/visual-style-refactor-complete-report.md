# 全站视觉风格化重构完整报告

日期：2026-05-22  
项目：Mr.Xia 个人小站 Astro 版本  
来源：基于 `visual-style-refactor-feasibility.md` 已确认内容重新整理  
状态：首轮实施报告，保留暂缓项

## 1. 报告结论

整个网站进行风格化、完整级别的视觉重构是可行的。当前项目已经完成 Astro 迁移，核心页面、内容集合、工具组件和文章增强能力都已经进入相对稳定的结构；本次重构不需要重新迁移内容，也不需要改 Worker、RSS、SEO 或文章 schema。

推荐路线是“设计系统级重构 + 轻量信息层级调整”。不要一次性替换全站 CSS；应先建立 token、主题、排版、组件边界、动效和文案规则，再按页面族群分阶段推进。首轮实施范围已经确定为“全局外壳 + 首页”。

总体判断：

| 维度 | 结论 |
| --- | --- |
| 技术可行性 | 高 |
| 视觉决策复杂度 | 中，主方向和首轮范围已经收敛 |
| 实现复杂度 | 中高 |
| 主要风险 | 全局 CSS 牵连、light / dark 双主题、移动端、交互页回归 |
| 推荐推进方式 | 多 phase、可回退、可阶段上线 |
| 首轮范围 | 全局外壳 + 首页 |
| 内容 schema | 暂不修改 |
| Worker / RSS / SEO | 暂不修改 |
| CI/CD | 首轮暂不强制新增；后续若改文件操作或加入截图回归，应补 workflow |

## 2. 项目现状

当前项目是完整 Astro 静态站点：

- Astro 配置和源码位于 `package.json`、`astro.config.mjs`、`tsconfig.json`、`src/`。
- 全局布局由 `src/layouts/BaseLayout.astro` 承载，引入 `src/styles/global.css`，并挂载 Header、Footer、DynamicBackground、SkipLink、TransitionIndicator。
- 内容集合在 `src/content.config.ts` 中定义，包括 blog、works、tools、updates。
- 主要页面包括 `src/pages/index.astro`、`src/pages/articles.astro`、`src/pages/articles/[...slug].astro`、`src/pages/articles/archive.astro`、`src/pages/works.astro`、`src/pages/works/tools.astro`、`src/pages/markdown-tool.astro`、`src/pages/about.astro`、`src/pages/new-post.astro`。
- 工具组件包括 `src/components/ToolsSection.astro`、`TimerWidget.astro`、`RandomSelector.astro`、`MarkdownToolWidget.astro`。
- 本地新建文章组件为 `src/components/NewPostForm.astro`，由开发环境的 `src/pages/new-post.astro` 启用。
- 文章增强能力在 `src/lib/article-enhancements/`，包括图片灯箱、标题锚点、目录、阅读进度、逐段渐显。
- 文章发布相关脚本包括 `scripts/publish-post.js`、`scripts/post-utils.js`、`scripts/markdown-utils.js`。
- 本地新建文章 API 位于 `tools/api-server.js`。
- Worker 入口为 `src/worker.ts`，文章阅读量通过 `src/lib/umami-view-counter.js` 代理 Umami。
- 静态资源和 legacy 兼容入口位于 `public/`。

当前视觉主要集中在 `src/styles/global.css`，该文件约 3167 行，包含 token、base、布局、组件、页面、工具、文章、响应式和动画。它是本次重构最大的影响面，也是最需要整理的地方。

## 3. 当前视觉问题

现有风格可以概括为暖色玻璃拟态：

- 橙、粉、青绿三色贯穿全站。
- `card`、`shadowbox`、`page-intro`、`blog-card`、`filter-nav`、`editor-panel` 等大量样式依赖大圆角玻璃面板。
- 标题、背景和按钮大量使用渐变与透明叠层。
- 全局已经存在页面过渡、按钮 ripple、列表入场、文章转场、灯箱、loading spinner。
- 内容密度中等，结构清楚，但视觉承载过度依赖卡片，节奏容易趋同。

新视觉不延续当前暖色玻璃拟态。旧风格作为历史版本保留，不作为后续视觉语言。

## 4. 核心设计方向

### 4.1 视觉 thesis

一个中性、直接、轻装饰的中文个人主页：保留博客的清爽阅读和个人站的入口感，用小圆角、统一中性色、适度留白、克制动效和手动 light / dark 主题切换建立稳定气质。首轮从全局外壳和首页开始，先把导航、主题切换、背景、首页标题和时间组件的基调定住。

### 4.2 风格采用

- 中文互联网个人主页的亲切结构。
- Fuwari 类博客的轻量、清爽、内容导向感，但不做得花哨。
- 中等内容密度。
- 小圆角、轻边框、低噪声背景。
- 中性色基础上的单一克制强调色。
- 卡片只用于确实需要框定的内容或交互。
- light / dark 双主题。
- 默认固定浅色，深色通过用户手动切换进入，不默认跟随系统主题。

### 4.3 风格避免

- 通用 SaaS。
- 二次元。
- 赛博朋克。
- 极简黑白。
- 过度玻璃拟态。
- 强科技蓝。
- 营销落地页。
- 过度装饰。
- 过度解释型文案。
- 卡片套卡片。
- 将工具页做成复杂 dashboard 或重型工作台。

## 5. 已确认产品定位

- 站点定位：混合形态。
- 导航顺序：文章、作品、工具、关于，保持现状。
- 主要访客：同学朋友、技术同行、潜在合作方、自己日常使用、搜索引擎访客都存在，不偏向单一人群。
- 首页职责：入口页，指引访问者进入文章、作品、工具、关于，不承担强营销或强叙事。
- 文章优先级：网站整体视觉效果与中文阅读体验同等重要。
- 工具定位：作品展示的一部分，不升级为主导航，不做严肃生产力工作台。
- 最近更新：保留，可以顺带重构。
- 全站搜索：可以保留，但不是首页重点。
- 作品主推：不突出单个主项目。
- 关于页：暂时不增加个人介绍。
- 文章归档：作为索引。
- Styleguide：内部维护，首轮暂缓，不公开强化。

## 6. 文案边界

全站文案必须短、直接、必要。

允许：

- 导航。
- 命名。
- 状态。
- 必要约束。
- 行动提示。
- 很短的识别性副标题。

不允许：

- “点击这里可以……”
- “这里提供了什么什么功能……”
- 为了解释设计而写在前端里的说明句。
- 把工具页、首页、tab、按钮变成说明书。
- 过度营销化、口号化或教程化的段落。

页面结构和首页/作品页文案可以重写，但不能破坏以上语言边界。

## 7. 视觉系统约束

- 色彩：偏中性、统一色系；不保留现有橙、青绿、粉作为硬约束。
- 强调色：暂不定稿；先在全局外壳和首页中进行页面试色，或后续使用其他 skill 专门探索。
- 主题：同时支持 light / dark。
- 默认主题：固定浅色，不默认跟随系统主题。
- 主题切换：用户手动切换，保存选择，避免首屏闪烁。
- 主题切换控件：桌面端和移动端分别设计，允许位置、尺寸和交互有所区别。
- 圆角：整体小圆角。
- 阴影：低阴影，避免大面积浮层感。
- 卡片：按功能使用，禁止卡片套卡片和滥用卡片。
- 背景：保留一定装饰性，但低噪声、不花哨、可降级。
- 首屏：暂不做 full-bleed，继续以站点主容器体系推进。
- 视觉素材：不要过分引入；暂不需要 hero 图片、生成图、项目截图、地图、代码片段作为主视觉。
- 图标：尽量少用；主题切换等清晰功能可以使用简单图标。
- 页面 logo：已有浏览器标签页 logo，页面上暂时不强化展示。

## 8. 排版与内容呈现

- 中文字体：混合方向，使用可靠、常见、质量稳定的字体，不用奇怪小众字体。
- Web Font：可以接受，但必须谨慎选择并控制加载成本。
- 英文、数字、计时器、代码字体：按场景决定。
- 正文：博客风格，不改成杂志、论文或笔记系统。
- 标题：中立，不追求过大冲击。
- 正文行距：不刻意加宽。
- Markdown 扩展块：按现有 Markdown 渲染能力处理，后续只统一风格。
- 代码块：整体保持接近 GitHub 的清晰风格；不预先锁定当前 highlight.js 配色，可以根据新风格判断是否只改容器与 spacing，或同步调整配色。
- 文章图片：当前体验较满意，暂不大动，只随新风格调整。
- 标签和分类：保持现状层级。

## 9. 首轮实施范围

首轮实施范围为“全局外壳 + 首页”。

首轮需要处理：

- Header。
- Footer。
- DynamicBackground。
- SkipLink。
- TransitionIndicator。
- 主题切换控件。
- 基础按钮、链接、tag、badge、输入框、面板、列表、分隔线样式。
- 首页主标题。
- 首页当前时间组件。
- 首页入口层级。
- 首页最近更新索引。
- 首页搜索的辅助入口地位。

首轮不作为前置处理：

- 工具页整体重构。
- legacy 视觉覆盖。

## 10. 首页策略

首页是入口页，不是营销页。

已确认要求：

- 当前时间组件保留，但允许调整布局位置和视觉层级。
- 主标题需要换掉，不继续使用“欢迎来到 Mr.Xia 的小站”。
- 新主标题必须在不同视口宽度下换行自然，不能出现别扭断行、单字孤行或挤压。
- 不展示主推文章。
- 不展示主推作品。
- 不直接强化工具快捷入口。
- CTA 按内容模块自然出现，不强行规定数量。
- 是否加入滚动叙事由复杂度决定，不为了动效而增加结构。
- 不增加更强个人介绍。
- 最近更新作为索引保留，不改成时间线。
- 首页可以有动效，但不得因此加入解释型描述文案。
- 搜索可以保留，但降级为辅助入口。

首页主标题候选：

1. `Mr.Xia`
2. `Mr.Xia 的个人主页`
3. `文章、作品和工具`
4. `写作与开发记录`
5. `Calvin Xia / Mr.Xia`

这些候选只是首轮文案起点，后续需要结合首页布局和响应式换行验证再定稿。

## 11. 文章体系策略

文章体系是核心体验之一。阅读体验不能因为视觉重构变差。

文章列表：

- 形态定位为博客流。
- 分类和标签筛选保留。
- 搜索建议下拉体验保留。
- 无结果状态可以增强，例如清空筛选、换关键词或推荐最近文章。
- 视觉上降低工具感，保持清楚的博客索引体验。

文章正文：

- 保持博客阅读体验。
- 控制中文行宽、标题层级、代码块、引用、表格、KaTeX 和图片样式。
- 桌面端目录继续右侧 sticky。
- 移动端目录保持顶部折叠。
- 阅读进度条保留并换成新风格。
- 阅读量保持现状，不额外强化。
- 文章元信息展示顺序保持现状。
- 返回入口顶部和底部各放一个。
- Giscus 评论保留，是唯一互动方式。
- Giscus light / dark 主题需要跟随站点主题自动切换，并尝试做视觉适配。
- 图片灯箱和现有图注体验只做风格同步，不大改交互。
- 分类、标签和阅读信息保持辅助层级，不抢正文注意力。
- 不增加专门的长文模式。

## 12. 作品页策略

作品页保持项目列表和工具集入口的角色，不做过度作品集包装。

已确认要求：

- 不强推单个作品。
- 不需要截图或封面。
- 不区分“主项目”和“小实验”。
- 不需要时间线或状态字段。
- 暂时不需要为未来更多作品预留过滤或分类。
- 作品页继续承接工具集入口。
- 项目标签、GitHub 链接、访问链接、APK 下载提示按实际内容和功能决定，不预先强行建立复杂规则。
- 项目条目可以从大圆角玻璃卡片改为更轻的档案条目、列表块或单层面板。
- 不用截图、封面、时间线制造额外视觉负担。

## 13. 工具页策略

工具页偏作品展示的一部分，不是独立产品工作台。

已确认要求：

- 工具页不进入主导航。
- 保持作品体系下的工具展示。
- 现有 tab 结构如果存在就保留，不重新发明导航。
- tab、工具标题和按钮不添加解释型描述语句。
- 计时器、随机选择器、Markdown 工具并列，不设单一最重要工具。
- Markdown 工具可以做得更接近编辑器界面，并可以加入编辑区与预览区同步滚动。
- 工具页输入内容使用 localStorage 保存即可。
- 随机选择器不需要更强结果仪式感，只跟随整体视觉风格调整。
- 不需要全屏或专注模式。
- 不需要键盘快捷键。
- 工具控件尺寸和密度保持中立。
- 工具状态反馈保持中立，不额外追求强动效。
- Markdown 工具不共享文章 Markdown 正文样式。
- 微信公众号/发布预览与站内文章是不同场景，应保持各自排版语境。
- 输入框、按钮、tab、结果区需要稳定尺寸，避免交互时布局跳动。

## 14. 关于页与更新日志

关于页：

- 暂时不增加个人介绍。
- 保留法律、隐私、免责声明和联系方式。
- 视觉上轻量，不做大段解释。

更新日志：

- 可以顺带纳入同一风格。
- 不需要成为首页主叙事。
- 可以作为索引或记录型内容保持轻量。

## 15. Legacy 策略

legacy 页面跟现在一样做跳转。

处理原则：

- 优先保证旧路径兼容和跳转正确。
- 不把旧静态页面纳入新视觉覆盖主线。
- 不默认对旧静态页面做视觉同步。
- 如果审计中发现旧页面仍直接承载有效内容，再单独确认迁移或轻量覆盖，不并入首轮默认范围。
- 验收重点是跳转、链接和兼容路径，不是新视觉一致性。

## 16. 文章发布流程策略

正式文章发布以 Obsidian/R2 CLI 流程为主。

### 16.1 CLI 发布链路

主流程：

- `npm run publish -- --dry-run <obsidian-post-dir>`
- `npm run publish <obsidian-post-dir>`

相关文件：

- `scripts/publish-post.js`
- `scripts/post-utils.js`
- `scripts/markdown-utils.js`

行为边界：

- 读取 `OKP_VAULT` 下的文章目录。
- 转换 `file/` 资源链接。
- 写入 `src/content/blog/*.md`。
- 通过 R2/S3 客户端上传本地资源。
- 完全保持目录命名、资源上传、`file/` 链接转换和输出文件名。
- 不修改文章 frontmatter schema。
- 不修改文章 slug、RSS、Worker 或 SEO。
- publish 流程中的 tags 可以为空或跳过，不要求非空。

### 16.2 `/new-post` 本地工具

`/new-post` 是开发环境隐藏辅助，不是正式主流程。

相关文件：

- `src/pages/new-post.astro`
- `src/components/NewPostForm.astro`
- `tools/api-server.js`

已确认要求：

- 纳入视觉重构，但不用过度打磨。
- 只做轻量界面统一和可用性检查。
- 继续只在开发环境启用。
- 隐藏在公开导航之外。
- 不需要草稿保存。
- 不保存 `NEW_POST_SECRET`。
- 不做 slug/frontmatter 预览。
- 不做重复文件提示增强。
- 不整理 dry-run 输出。
- 不新增前端发布说明。

### 16.3 Markdown 工具边界

Markdown 工具保持公众号/HTML 工具定位，不与站内文章发布流程合并。

相关文件：

- `src/components/MarkdownToolWidget.astro`
- `src/scripts/markdown-renderer.ts`

边界：

- HTML 渲染工具中的 tag 非空校验可以保留。
- 该校验不扩散到 CLI 发布链路。
- Markdown 工具预览不共享文章正文样式。
- 微信公众号/发布预览与站内文章排版分开维护。

## 17. 动效与交互

已确认原则：

- 当前页面转场保留，并可以在新视觉系统下进一步打磨。
- 按钮 ripple 可以保留，但需要纳入统一动效 token。
- 动效显眼程度保持中立，不刻意强存在感，也不完全隐藏。
- 是否加入滚动触发、sticky 叙事或轻微深度效果，由页面复杂度和实际收益决定。
- 文章正文优先安静。
- 首页和列表状态可以稍微有层级动效。
- 文章页切换可以保留方向感。
- 工具页结果不需要额外动效，只保留必要状态反馈。
- 桌面 hover 可以更丰富。
- 移动端简化处理。
- reduced-motion 至少要避免眩晕、阻塞或不必要的大幅运动。
- loading indicator 保留，并随新视觉统一。

## 18. 可访问性与性能

基础要求：

- WCAG AA 对比度尽可能满足，尤其是正文、按钮、链接、输入框、标签、代码块和 dark mode。
- dark mode 不只是反色，需要单独检查文字、边框、按钮、代码块、评论、灯箱和表单。
- 保留 focus ring。
- 支持键盘焦点、skip link 和合理的 tab 顺序。
- 移动端常见宽度必须可用，避免横向滚动、遮挡、文字挤压或按钮过小。
- 移动端导航触控目标应不小于 44px。
- 浏览器兼容以现代浏览器为准。
- 可以为了可访问性调整颜色和视觉层级。
- 对低端设备性能的取舍保持中立，但不能为了装饰造成明显卡顿。
- 字体谨慎，减少 blur 和大面积阴影，避免重型视觉资产。
- 暂时不做图片 AVIF/WebP 多尺寸处理，因为当前不计划引入重型图片资产。

暂缓项：

- 视觉回归检查首轮先不急着加入。
- `styleguide.astro` 首轮先不急着产出。

## 19. CSS 与依赖策略

建议逐步从 `src/styles/global.css` 中整理逻辑层，而不是一次性大拆。

建议 CSS 分层：

- `tokens.css`
- `base.css`
- `layout.css`
- `components.css`
- `article.css`
- `tools.css`
- `pages.css`
- `utilities.css`

是否拆文件可以根据实现阶段决定。即使仍由 `global.css` 汇总，也应先建立清晰的逻辑层。

依赖策略：

- 接受新增 npm 依赖，包括图标库、字体工具、截图测试或视觉回归相关工具。
- 新增依赖必须有明确用途。
- 每个依赖需要记录用途、替代方案和移除成本。
- 不为了“看起来高级”随意增加依赖。
- 图标库只在确实能改善功能识别时引入。
- 视觉回归依赖首轮暂缓。

class 策略：

- 是否保留现有 class 名视情况而定。
- 公共语义 class 可保留以降低回归。
- 明显绑定旧视觉或阻碍重构的 class 可以替换。
- 不能为了保留 class 牺牲新系统边界。

## 20. 可行性拆解

### 20.1 视觉换肤级重构

可行性：高  
风险：中  
适合目标：保持现有页面结构，只换颜色、字体、圆角、背景、阴影、按钮、卡片和动效节奏。

优点：

- 工作量较可控。
- 不影响内容集合、路由、SEO、Worker、RSS。
- 可以快速得到明显的新视觉。

限制：

- 如果继续沿用大量卡片和玻璃面板，风格变化可能只是换皮。
- 直接替换变量不能解决全局 CSS 耦合问题。

### 20.2 设计系统级重构

可行性：高  
风险：中高  
适合目标：重新定义 token、布局节奏、页面模板、组件状态、文章排版、工具界面和动效。

优点：

- 能建立可长期维护的视觉系统。
- 可减少重复样式和卡片泛滥。
- 后续新增页面更容易保持一致。

限制：

- 需要逐页验证移动端和交互状态。
- 需要处理 `global.css` 的分层和迁移。
- 双主题需要同时覆盖基础组件、文章、工具和评论。

### 20.3 信息架构加视觉重构

可行性：中高  
风险：高  
适合目标：不仅换视觉，还重新定义首页叙事、文章入口、作品展示、工具层级和搜索方式。

优点：

- 最接近全站重新设计。
- 可以让个人站的定位更鲜明。

限制：

- 容易扩大范围。
- 会影响页面文案、导航策略和用户路径。
- 更需要移动端检查和回归验证。

推荐采用第二种为主：设计系统级重构，并做轻量信息层级调整。

## 21. 粗略工作量

| 范围 | 预估 |
| --- | --- |
| 只做视觉换肤 | 2 到 4 个工作日 |
| 设计系统级重构 + light / dark | 7 到 11 个工作日 |
| 视觉 + 轻量信息层级调整 + 截图回归 | 10 到 14 个工作日 |
| 全站多 phase 完整改造 | 12 到 18 个工作日 |

估算默认单人推进，不包含大量新增插图、摄影、复杂动画或视觉资产管线。由于目标是全站改造，不建议只做一个原型页后停止；可以先做代表性页面校准，但后续要进入全站分阶段落地。

## 22. 分阶段实施规划

### 阶段 0：页面审计与首轮范围固化

目标：

- 固化首轮范围：全局外壳 + 首页。
- 将强调色、视觉回归检查、`styleguide.astro` 首轮建设作为暂缓项记录。
- 建立页面清单和状态清单：空状态、加载、搜索结果、无结果、错误、评论、图片灯箱、移动端导航、工具 tab、表单状态。
- legacy 页面按当前方式保持跳转，阶段 0 只确认旧路径清单和跳转完整性。
- 固化文章发布流程边界：CLI 为主，`/new-post` 为开发环境隐藏辅助，Markdown 工具为公众号/HTML 工具。
- 建立多 phase 范围切分，明确每个阶段是否可独立上线、如何回退、如何验证。
- 确认新增依赖候选清单。

产出：

- 已确认问题记录。
- 全站页面重构范围表。
- legacy 跳转清单和验证方式。
- 文章发布流程保留现状、轻量纳入 `/new-post` 视觉统一的结论。
- 多 phase 执行切分。
- 新增依赖候选与取舍理由。
- 首页主标题候选和响应式换行检查要求。
- 不改动范围说明。

### 阶段 1：设计系统与主题基础

目标：

- 定义 light / dark 双主题 token：背景、文字、边框、强调色、链接、代码块、按钮、输入框、状态色。
- 默认主题固定为浅色；深色由用户手动切换，不默认跟随系统主题。
- 定义字体栈。
- 定义小圆角、轻边框、低阴影、间距、z-index、动效时长。
- 定义卡片使用规则。
- 定义低噪声装饰背景规则。
- 定义桌面端和移动端各自的主题切换策略。

### 阶段 2：站点外壳与基础组件

目标：

- 重构 Header、Footer、DynamicBackground、SkipLink、TransitionIndicator。
- 添加或改造主题切换控件。
- 优化桌面和移动端导航。
- 统一页面转场、按钮 ripple、loading indicator 和 reduced-motion 行为。
- 建立基础按钮、链接、tag、badge、输入框、面板、列表和分隔线样式。

重点：

- Header 不抢首屏预算。
- 移动端导航触控目标不小于 44px。
- 焦点状态必须清晰。
- 页面 logo 不额外强调。
- 组件不写说明性文案。
- hover 只作为桌面增强。

### 阶段 3：首页和内容入口

目标：

- 保持首页作为站点入口。
- 保留并重排当前时间组件。
- 替换首页主标题并验证响应式换行。
- 减少重复 CTA 和说明文字。
- 让文章、作品、工具、最近更新之间有更清晰的层级。
- 保留最近更新作为索引。
- 搜索降级为辅助入口。

可能改动：

- 首页 hero 从大圆角玻璃卡片改为小圆角或无卡片入口区域。
- 最近更新从卡片流改成更轻的列表或索引。
- 不新增工具主导航。
- 不添加主推文章、主推作品或首页工具快捷重点区。
- 首页动效只服务层级、进入状态或轻微反馈。

### 阶段 4：文章体系

目标：

- 重构文章博客流、筛选、搜索建议、无结果状态、归档页。
- 重构正文排版、目录、代码块、引用、表格、KaTeX、评论区。
- 文章图片与灯箱只做风格同步。
- 顶部和底部都提供返回入口。

重点：

- 中文阅读行宽稳定。
- 标题层级清楚。
- 桌面右侧 sticky 目录和移动端顶部折叠目录保持。
- 阅读进度条保留并换风格。
- Giscus 跟随 light / dark 自动切换。
- 代码块接近 GitHub 风格，但配色和容器可调整。
- 不增加长文模式。

### 阶段 4.5：文章发布与本地创作工具

目标：

- `/new-post` 只做视觉统一和基础可用性检查。
- 不改写文件逻辑。
- 不做草稿保存。
- 不新增发布说明。
- 保持 CLI 发布逻辑稳定。
- 保持 Markdown 工具与站内文章发布流程分离。

如果后续改动文件写入、发布计划、payload 校验或上传逻辑，需要补测试和 CI/CD workflow。

### 阶段 5：作品和工具体系

目标：

- 作品页升级为轻量项目档案。
- 工具页保持作品体系下的展示和使用。
- Timer、Random Selector、Markdown Tool 并列展示。
- 保留现有 tab。
- Markdown 工具增强编辑体验和同步滚动。
- 工具输入使用 localStorage 保存。
- 顺带统一更新日志页面视觉。

重点：

- 不做大型营销 hero。
- 不升级为独立产品页。
- 工具控件、输入、结果区稳定。
- 不加截图、封面、时间线、状态字段或未来过滤。
- 不为 tab 或工具说明增加描述性语句。

### 阶段 6：legacy 跳转与辅助页面

目标：

- 保持 legacy 跳转。
- 验证旧路径和兼容文件没有被破坏。
- 不默认视觉同步旧静态页面。
- `styleguide.astro` 首轮暂缓，后续按 token 稳定情况再评估。

### 阶段 7：质量验证

最低验证：

- `npm run build`
- `npm test`
- 桌面宽度：首页、文章列表、文章正文、作品页、工具页、关于页、更新日志。
- 移动宽度：至少覆盖常见手机宽度，重点检查 375px、480px、768px。
- light / dark 两套主题。
- dark mode 对比度。
- 键盘焦点。
- 搜索建议。
- 工具 tab。
- 评论。
- 图片灯箱。
- 返回链接。
- reduced-motion。
- 浏览器 console。
- 现代浏览器。

后续可选增强：

- Playwright 截图检查。
- 视觉回归基准。
- `styleguide.astro` token 和组件状态展示。

### 阶段 8：分阶段上线与收口

目标：

- 每个 phase 完成后保证可构建、可回退、可独立验收。
- 如果阶段允许独立上线，记录上线范围和剩余未覆盖页面。
- 全站改造完成后做一次整体视觉一致性审计。

重点：

- 不因“慢慢来”而停留在原型页。
- 阶段边界围绕用户可见页面族群，不随意按 CSS 文件拆。
- 每个阶段都保留全局语言边界。

## 23. 风险矩阵

| 风险 | 等级 | 原因 | 缓解方式 |
| --- | --- | --- | --- |
| 全局 CSS 牵连过大 | 高 | 多页面共用旧 class | 先抽 token 和组件层，再逐页替换 |
| light / dark 不一致 | 高 | 双主题需要覆盖背景、文字、边框、代码块、表单、giscus、灯箱 | 主题 token 先行，每阶段同时验收 |
| 文章正文回归 | 高 | Markdown、代码、表格、图片、KaTeX、目录、灯箱依赖样式 | 建立正文样例页和长文检查清单 |
| 卡片继续滥用 | 中高 | 现有视觉高度依赖卡片和玻璃面板 | 明确单层容器原则，优先列表、分隔线、网格和留白 |
| 工具页可用性下降 | 中高 | 视觉化可能影响输入、按钮、tab、预览区效率 | 工具页保持可用优先，控件和状态稳定 |
| 误改文章发布流程 | 高 | 发布链路涉及文件写入、R2 上传、资源链接转换和本地 API | CLI 主流程保持现状，`/new-post` 只做轻量视觉统一 |
| 文件操作改动缺少 CI | 高 | 仓库要求文件操作功能配套自动化校验 | 改 `createPostFile`、发布计划或上传逻辑时同步补测试和 workflow |
| 本地创作页暴露敏感信息 | 中高 | `/new-post` 使用 `NEW_POST_SECRET` | 不做草稿保存，不保存 secret |
| 首页标题响应式断行难看 | 中 | 新标题可能在移动端孤字或挤压 | 在 375px、480px、768px 和桌面宽度检查 |
| Markdown 工具与文章样式混同 | 中 | 公众号预览和站内文章是不同场景 | 工具预览和文章正文分开定义 |
| localStorage 状态处理不一致 | 中 | 工具输入保存可能产生过期、空值或跨 tab 行为差异 | 建立统一 key、初始化、清空和错误处理策略 |
| 移动端首屏拥挤 | 中高 | Header、hero、CTA、时间组件可能挤占首屏 | 先定义移动端首屏预算 |
| 动效过度 | 中 | 当前已有转场、ripple、reveal、灯箱 | 统一 motion token，文章正文保持安静 |
| reduced-motion 处理不足 | 中 | 不能因动效产生眩晕或阻塞 | 关闭大幅位移、强滚动动效和非必要循环动画 |
| 桌面 hover 与移动端行为不一致 | 中 | 桌面 hover 更丰富，移动端简化 | hover 只作为增强，不承载必要信息 |
| 性能变差 | 中 | 双主题、字体、背景纹理、动效可能增加负担 | 字体谨慎，减少 blur 和大阴影，避免重型资产 |
| 新依赖膨胀 | 中 | 已接受新依赖，可能增加维护成本 | 每个依赖必须记录用途、替代方案和移除成本 |
| phase 边界失控 | 中高 | 多阶段容易半新半旧 | 每个 phase 定义范围、验收、回退、上线状态 |
| 旧 class 与新系统冲突 | 中 | 旧 class 可能绑定旧视觉 | class 保留按风险判断 |
| dark mode 对比不足 | 中高 | 深色主题不是简单反色 | 单独检查 dark mode 对比度 |
| 文案变啰嗦 | 中 | 重构时容易补解释型提示 | 只保留导航、状态、必要提示和行动 |
| 品牌方向摇摆 | 中 | 强调色和细节仍可探索 | 把未定问题限制在页面试色和后续 skill |
| 可访问性下降 | 中高 | 低对比、隐藏焦点、图像文字叠加 | WCAG 对比、键盘导航、触控尺寸作为硬约束 |
| legacy 跳转失效 | 中 | `public/` 中仍有旧 URL 和兼容入口 | 阶段验收旧路径和链接 |

## 24. 验收标准

一次成功的全站视觉重构应该满足：

- 访客在 5 秒内能看出这是 Mr.Xia 的个人站，而不是通用模板。
- 首页、文章、作品、工具四类页面有同一套视觉 DNA，但各自任务清楚。
- 首页主标题在常见视口宽度下换行自然。
- 文章正文比现在更好读，而不是更花。
- 文章列表是清楚的博客流。
- 筛选、搜索建议、无结果状态保持可用。
- 文章目录、阅读进度、阅读量、Giscus 和返回入口符合已确认行为。
- Giscus 跟随站点主题自动切换。
- 文章发布流程没有被视觉重构隐性改变。
- `/new-post` 保持本地开发环境入口属性，不做草稿保存，不暴露或保存密钥。
- publish 流程允许 tags 跳过。
- 工具页比现在更稳定，同时仍像作品体系的一部分。
- 工具页保留 tab，三个工具并列。
- Markdown 工具支持更好的编辑体验与同步滚动。
- Markdown 工具预览和站内文章正文保持场景区分。
- 作品页不通过截图、封面、主次项目、时间线、状态字段或过滤系统增加复杂度。
- legacy 页面保持当前跳转兼容。
- 全站减少无意义卡片，不出现卡片套卡片。
- light / dark 两套主题完整、可读、可手动切换。
- 默认固定浅色。
- dark mode 单独检查对比度。
- 页面转场、按钮 ripple、loading indicator 保留并统一新风格。
- 桌面 hover 是增强体验，移动端简化后仍能完整操作。
- reduced-motion 下没有大幅位移、强滚动动效或阻塞性动画。
- 前端文案没有无意义解释。
- 移动端没有横向滚动、遮挡、文字挤压或按钮过小。
- 键盘焦点、aria 状态、skip link、focus ring 不退步。
- WCAG AA 尽可能满足。
- 如果改动文件操作功能，必须有对应测试和 CI/CD 校验。
- 页面结构和文案可以重写，但全局语言边界不被破坏。

## 25. 暂缓决策项

以下三项暂时搁置，不作为首轮实施前置条件：

1. 新视觉强调色：先根据页面试色，或后续使用其他 skill 进行专门尝试。
2. 视觉回归检查：首轮先不急，等第一批页面稳定后再决定。
3. `styleguide.astro` 内部页：首轮先不急，待 token 和基础组件稳定后再评估。
